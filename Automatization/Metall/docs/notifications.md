# Уведомления — система цеха металлоконструкций

## Реализованные

| # | Триггер | Кому | Статус |
|---|---------|------|--------|
| 1 | Рабочий поставил БЛОК на задачу | Мастер + Админ | ✅ Готово |
| 2 | Мастер снял БЛОК | Рабочий (кто заблокировал) | 🔧 В работе |
| 3 | Все детали ПЛАЗМА + ПИЛА по элементу выполнены | Сборщик | 📋 Запланировано |

## Запланировано к реализации

| # | Триггер | Кому | Примечание |
|---|---------|------|-----------|
| 4 | Рабочий не выходил на связь за смену | Мастер | Нет действий в боте за день — автоматическая проверка вечером |
| 5 | Просрочена дата план | Мастер + Админ | Вечерний отчёт по всем просроченным позициям |

## Решения по блокировкам

Уведомления №7 и №8 из первоначального списка (нет материала, перегрузка оборудования)
реализуются через тот же механизм блокировки — рабочий ставит БЛОК с соответствующей причиной.
Отдельного типа уведомления не нужно.

## Зависимости между этапами

**Текущее решение:** логика в коде (без отдельной таблицы).

Правило: перед тем как сборщик видит задачу по элементу — проверяем SQL:
все ли позиции ПЛАЗМА + ПИЛА по этому `element` имеют `status = 'ВЫПОЛНЕНО'`?

**Когда переходить на таблицу `stage_dependencies`:**
при появлении второго правила, например ПОКРАСКА ждёт СВАРКУ.
