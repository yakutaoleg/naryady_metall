## 🔴 КРИТИЧНО — сделать в первую очередь

### 1. Нет бэкапов PostgreSQL
Директории `/root/backups/` не существует, cron-задач на дамп нет. Если VPS упадёт — вся история нарядов, сотрудники и проекты потеряны без восстановления.

**Фикс (15 минут):**
```bash
mkdir -p /root/backups
# Добавить в crontab на VPS:
0 3 * * * pg_dump naryady | gzip > /root/backups/naryady_$(date +%Y%m%d).sql.gz
find /root/backups/ -mtime +14 -delete
```

---

### 2. Race condition — двойное нажатие (`bot_remote.py:805, 875`)
Если рабочий дважды быстро нажмёт ВЫПОЛНЕНО — бот подтвердит оба раза. Если два мастера одновременно снимают один блок — рабочий получит два уведомления.

**Фикс:** проверять `rowcount` после UPDATE, при 0 — `show_alert("Задача уже отмечена")`.

---

### 3. IDOR — доступ к чужим задачам (`bot_remote.py:915, 839`)
Обработчики `block_ask:` и `drawing:` не проверяют что задача принадлежит текущему рабочему. Перебором `block_ask:1`, `block_ask:2`... любой пользователь получает позиции/элементы всех задач в системе и может скачать чужие чертежи из Google Drive.

Для сравнения — в `done:` (строка 791) проверка уже есть: `task['executor'] != worker_name`.

**Фикс:** добавить ту же проверку в `block_ask:` и `drawing:`.

## 🟡 ВАЖНО — сделать в ближайшем деплое

### 4. Синхронизация зависимостей без транзакции (`sync_remote.py:129`)
DELETE + INSERT не атомарны. Если процесс упадёт между ними — зависимости удалены, сборщики получат задачи без ожидания деталей.

**Фикс:** обернуть DELETE+INSERT в транзакцию.

---

### 5. Блокировка event loop при скачивании чертежа (`bot_remote.py:850`)
`_download_drawing()` — синхронная функция в async контексте. Пока качается тяжёлый PDF — все пользователи бота "замерзают".

**Фикс:**
```python
pdf_bytes, name = await asyncio.get_event_loop().run_in_executor(
    None, _download_drawing, file_id
)
```

---

### 6. Кнопка "Синхронизировать" без cooldown (`bot_remote.py:680`)
Мастер может нажать 50 раз подряд — 50 параллельных процессов синхронизации на VPS.

**Фикс:** cooldown 60 секунд через `context.bot_data`.

---

### 7. Текст ошибки уходит пользователю (`bot_remote.py:1013`)
При ошибке сканирования папки полный текст исключения (включая пути к БД) отправляется в чат.

**Фикс:** логировать `e`, пользователю — только "Ошибка. Обратитесь к администратору."

---

### 8. Роль мастера кешируется без перепроверки
Если убрать роль в БД — сотрудник остаётся мастером в боте до перезапуска.

**Фикс:** перепроверять роль из БД при master-действиях, или TTL на `user_data`.

---

### 9. N+1 запросы в `_deps_ready` (`bot_remote.py:172`)
При 20 задачах у сборщика с зависимостями — до 200+ запросов к БД на одно нажатие "Задачи".

**Фикс:** один запрос `WHERE (sheet_name, position) IN (...)` вместо цикла.

---

### 10. Масштабирование — нет мультитенантности
БД не изолирована по организации. При появлении второго заказчика — нужна колонка `tenant_id` или отдельная схема. Сейчас заложить несложно, потом переделывать дорого.

## 🟢 UX — быстрые wins

### 11. Сумма дня — в главное меню
Рабочий должен видеть заработок не заходя в подменю. Добавить одну строку в `show_menu`:  
`💵 Заработано сегодня: 13.39 руб`

### 12. Выработка в тоннах для мастера
`total_weight` уже есть в БД, просто не суммируется. Добавить `SUM(total_weight)` в `get_earnings()`.

### 13. Прогресс проекта в карточке
Сейчас: только статус и время синхронизации.  
Нужно: `47 из 120 позиций выполнено · 3 в блоке`

### 14. Возврат к списку блоков после снятия
После `unblock` мастер теряет контекст — нужна кнопка "← К блокам" или автообновление списка.

### 15. Минимальная длина комментария при блоке
Сейчас принимается "." или пробел. Минимум 10 символов снизит мусорные блоки.

### 16. Дата блокировки видна рабочему
Рабочий видит заблокированные задачи, но не знает сколько дней они висят. Добавить дату в карточку.
